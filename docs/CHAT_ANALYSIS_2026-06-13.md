# Аналитический отчёт по обсуждению GOART и AWB

Дата фиксации: 2026-06-13.

Этот документ собирает идеи, решения, замечания и риски, которые были сформулированы в текущем обсуждении проекта GOART. Цель файла — не заменить код, а сохранить проектную память, чтобы при следующих правках ничего не потерять.

---

## 1. Главная цель проекта

GOART рассматривается как русскоязычная браузерная игра про рабочий день в типографии.

Ключевая цель:

- довести проект до стабильного состояния;
- сохранить все игровые идеи;
- восстановить потерянные или повреждённые части кода;
- оформить документацию в GitHub;
- не ломать текущую рабочую версию;
- в будущем использовать проект как основу для мобильной веб-игры или Android WebView-приложения.

---

## 2. Основная игровая задумка

Игрок находится в типографии и перемещается между комнатами. По карте ходят NPC. Каждый NPC может:

- выдавать квест;
- мешать игроку;
- помогать;
- снижать или повышать стресс;
- запускать диалог;
- участвовать в сюжетном продвижении.

Игровой путь сейчас строится вокруг трёх задач:

1. Получить и выполнить цветопробу.
2. Выполнить лак.
3. Дойти до Виктора и пройти финальный блиц.

## 3. Карта и структура мира

В обсуждении зафиксирована карта из 13 локаций:

```text
0  Офис
1  Коридор
2  Склад 1 эт
3  ПР. цех дверь
4  Аквариум
5  Коридор 2
6  Дверь на 2 эт
7  Проход в произв.
8  Производство
9  Лак
10 Паллеты
11 Склад 2 эт
12 Кабинет босса
```

Каждая комната имеет:

- `id`;
- `name`;
- `icon`;
- `doors`;
- `desc`;
- `limit`.

Важная идея: локации не просто декоративные. Они влияют на движение, доступность действий, маршруты NPC и ограничения по количеству персонажей.

## 5. Обсуждённая логика движения NPC

### Обычные NPC

Функция `startNpcMovement(npc)` должна:

- запускать индивидуальный таймер движения;
- не двигать NPC до спавна;
- не двигать Марину, если для неё предусмотрена отдельная логика;
- выбирать доступную комнату из `home`;
- учитывать лимит комнаты;
- учитывать особое правило аквариума и Марины;
- при заторах использовать туннелирование.

### Туннелирование

Туннелирование — это игровая условность: если все комнаты по маршруту заняты, NPC не застревает навсегда, а проходит через переполненную комнату дальше.

Важно: даже при туннелировании нельзя нарушать правило Марины в аквариуме.

### Виктор

Виктор должен иметь отдельную логику патруля:

- появляется в офисе;
- идёт по маршруту к кабинету;
- после первого прохода запускает циклический патруль;
- ходит туда-обратно;
- между обходами делает паузу `patrolInterval` или 60 секунд по умолчанию.

## 6. Обсуждённые правки к коду

В текущем обсуждении были разобраны и рекомендованы следующие правки:

### 6.1. Блок старта и рендера

Нужно убрать дубли `renderFooter()` и не вызывать несуществующую `renderControls()`.

Финальный `renderAll()` должен быть таким:

```js
function renderAll() {
  renderMap();
  renderQuests();
  renderFooter();
}
```

### 6.2. `resetGame()`

Рекомендуется защищаться от отсутствующего DOM-элемента `player-name`:

```js
const playerNameEl = document.getElementById('player-name');
if (playerNameEl) playerNameEl.innerText = n;
```

### 6.3. `shuffle()`

Должна быть только одна функция `shuffle()` во всём проекте.

### 6.4. `startQuizBOSS()`

В коде были найдены дубли функции `startQuizBOSS()`. Нужно оставить одну финальную версию.

### 6.5. Утилиты

Функция `randDialog(npc)` должна быть устойчива к отсутствующим `prob2`, `prob3` или `dialog2`, `dialog3`.

## 7. Важные риски потери кода

По текущему анализу главный риск — не сама игровая идея, а повреждение структуры кода при ручных вставках.

Типичные признаки повреждений:

- дубли функций;
- незакрытые комментарии;
- повторное объявление одной функции ниже по файлу;
- HTML-шаблоны внутри JS потеряли обратные кавычки;
- DOM-элементы вызываются из JS, но отсутствуют в HTML;
- `js/main.js` и корневой `main.js` могут конфликтовать по смыслу;
- документация и фактический код могут расходиться.

## 8. AWB — отдельная, но связанная идея

AWB — Adaptive Web Browser / Адаптивный веб браузер.

Это не сама игра, а потенциальная Android WebView-обёртка для GOART и других сайтов.

Зафиксированные требования:

- URL по умолчанию: `https://sunpole.github.io/GOART/`;
- возможность менять URL;
- переключаемый Landscape;
- все чекбоксы выключены по умолчанию;
- режим без масштаба;
- режим без прокруток;
- режим без выделения и копирования текста;
- светлая/тёмная тема;
- таймер нахождения на странице;
- описание проекта внутри приложения;
- поддержка старых Android, особенно Android 7 на Samsung Galaxy Note 5.

## 9. Два пути развития интерфейса

В обсуждении были сформулированы два возможных пути.

### Путь A: APK / WebView / Landscape / Full HD

Если получится собрать Android APK, можно развивать интерфейс как более широкую игровую сцену с принудительным или переключаемым Landscape.

### Путь B: обычный мобильный веб

Если APK собрать сложно, проект нужно вести как мобильный сайт шириной около 360–412 CSS px.

Вывод: пока нет стабильной APK-сборки, основной веб-интерфейс должен оставаться адаптивным и не зависеть от Full HD.

## 10. Правильная стратегия восстановления

Рекомендуемый порядок:

1. Не переписывать всё сразу.
2. Сначала зафиксировать документацию.
3. Затем сделать резервную ветку.
4. Затем проверить `index.html`, `main.js`, `js/main.js`.
5. Исправить только критические ошибки запуска.
6. Проверить GitHub Pages.
7. После стабильной версии начать маленькие рефакторинги.
8. Только потом делить проект на модули.
