# Архитектура GOART

## Текущая архитектура

Сейчас проект устроен максимально просто:

```text
GOART/
├── index.html
├── main.js
├── README.md
├── PROJECT_NOTES_AWB_GOART.md
├── docs/
├── js/
└── img/
```

`index.html` содержит:

- HTML-разметку;
- встроенные CSS-стили;
- основные DOM-контейнеры;
- подключение корневого `main.js`.

`main.js` содержит почти всю игровую логику:

- данные комнат;
- данные NPC;
- квесты;
- состояние игры;
- утилиты;
- спавн NPC;
- движение NPC;
- старт и сброс игры;
- рендер карты;
- рендер квестов;
- рендер футера;
- обработку перемещений;
- действия в комнатах;
- обработку встреч с NPC;
- модальные окна событий;
- финальный блиц босса;
- справку.

`js/main.js` требует отдельной проверки. Если он содержит только glue-code вида `window.onload = () => { resetGame(); };`, его нельзя считать основной логикой. Важно не допустить двойного запуска игры, если в будущем будет подключено несколько JS-файлов.

## Основные DOM-элементы

В `index.html` используются или должны использоваться:

```text
#header
#main-wrap
#map
#questlog
#footer
#eventBox
#eventPortrait
#eventText
#eventOptions
#descModal
#legendCont
```

Код в `main.js` должен обращаться только к тем DOM-элементам, которые реально существуют в `index.html`.

## Важное правило стабильности

Перед обращением к DOM-элементу желательно либо гарантировать его наличие в HTML, либо использовать защитную проверку:

```js
const el = document.getElementById('some-id');
if (el) {
  el.innerText = 'Текст';
}
```

Это особенно важно для проекта, который часто меняется через генерацию и ручные правки.

## Логические слои проекта

Даже если всё пока находится в одном `main.js`, архитектурно проект уже делится на слои.

### 1. Данные

```text
ROOMS
NPCS_FULL
QUESTS
BOSS_QUIZ
```

Этот слой не должен напрямую менять DOM.

### 2. Состояние

```text
gameState.player
gameState.npcs
gameState.dialogOpen
```

Состояние должно быть единым источником правды: где игрок, какие квесты выполнены, какие NPC появились, что в инвентаре.

### 3. Утилиты

```text
deepClone
randDialog
shuffle
```

Утилиты должны быть объявлены один раз.

### 4. Спавн и движение NPC

```text
startAllNpcSpawns
startNpcMovement
startViktorLogic
movePatrol
```

Здесь важно контролировать таймеры, чтобы при перезапуске игры не появлялись старые интервалы.

### 5. Рендер

```text
renderAll
renderMap
renderQuests
renderFooter
```

Рендер должен только отображать состояние, а не создавать скрытые побочные эффекты.

### 6. Игровые действия

```text
moveTo
makeProba
makeLak
checkEvents
```

Этот слой меняет `gameState` и вызывает рендер.

### 7. Модальные события

```text
showEvent
showEventNPC
eventAction
```

Модалки должны корректно ставить и снимать флаги `dialogOpen` и `player.busy`.

### 8. Финал

```text
startQuizBOSS
BOSS_QUIZ
```

Финальный блиц должен быть объявлен один раз и иметь понятную логику победы/ошибки.

## Текущие архитектурные проблемы

1. Весь код находится в одном большом `main.js`.
2. CSS встроен в `index.html`.
3. В рабочих версиях кода встречались дубли `shuffle()`, `renderFooter()` и `startQuizBOSS()`.
4. В одной из версий был вызов `renderControls()`, хотя управление уже встроено в `renderFooter()`.
5. Нет отдельного слоя данных, рендера и игровой логики.
6. Не хватает автоматической проверки целостности данных.
7. Нет тестового режима для быстрой проверки квестов и финала.
8. Возможна путаница между `main.js` в корне и `js/main.js`.
9. Есть риск накопления старых таймеров NPC при перезапуске игры.
10. Возможны обращения к DOM-элементам, которых нет в HTML.

## Рекомендуемый порядок рефакторинга

1. Сначала исправить ошибки запуска в текущем виде.
2. Зафиксировать рабочую версию коммитом.
3. Создать резервную ветку или тег релиза.
4. Убрать дубли функций.
5. Защитить обращения к DOM.
6. Проверить полный игровой путь.
7. Только потом переносить CSS в отдельный файл.
8. Затем отделить данные от логики.
9. Затем отделить рендер от игровой логики.
10. Затем убрать устаревшие или неиспользуемые файлы.

---

## Почему не нужно сразу переписывать всё

Проект уже имеет рабочую игровую идею, карту и данные. Главный риск — потерять текущую логику при слишком резком рефакторинге.

Правильный подход:

```text
сначала стабилизация → потом документация → потом маленькие патчи → потом рефакторинг
```
